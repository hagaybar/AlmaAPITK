# Release 0.4.0 — design

**Date:** 2026-05-10
**Owner:** Hagay Bar
**Status:** Approved (brainstorming complete)
**Predecessor release:** `0.3.1` (2026-04-27)

## Summary

Cut and publish `almaapitk 0.4.0` to PyPI. The release bundles ~168
commits since `v0.3.1`: a new `Configuration` domain, `Admin.Sets` full
CRUD, a large `Users` coverage push, and substantial client
architecture work (persistent Session, retry, region map, error
taxonomy, `iter_paged`, context-manager, `AlmaResponse.data` cache).
All changes are additive — no breaking signatures.

Publish is manual (`poetry build` + `poetry publish`), gated by a
TestPyPI dry-run, and walked through a `release/0.4.0` branch so the
version bump and CHANGELOG finalize land as a single reviewable unit.

## Decisions made during brainstorming

| Decision | Choice | Rationale |
|---|---|---|
| Scope | Finish `users-requests` chunk first, then release | Users coverage block ships as one coherent story; small wait |
| Version | `0.4.0` (minor) | Additive features, matches SemVer; preserves `1.0.0` for an intentional stability declaration |
| Validation bar | Standard | Smoke + unit/logging/client suites + R10 regression-smoke + TestPyPI install gate |
| Publish mechanics | Manual (`poetry publish`) | Single-author project; CI workflow filed as follow-up |
| Release branch | `release/0.4.0` PR'd to `main` | Reviewable unit for version bump + CHANGELOG; clean rollback target |

## Sequence of operations

Linear, one gate per phase. Nothing is irreversible until step 7.

1. **Finish `users-requests` chunk.**
   `/chunk-run-test users-requests` → review SANDBOX results → squash-merge
   PR to `main` → `scripts/agentic/chunks complete users-requests --pr-url <URL>`.

2. **Open release branch.**
   `git checkout main && git pull && git checkout -b release/0.4.0`.

3. **Finalize CHANGELOG** (see CHANGELOG plan below).

4. **Bump version.**
   `pyproject.toml` only — `version = "0.3.1"` → `"0.4.0"`. Verified to
   be the single version constant in the repo.

5. **Run validation suite** (see Validation suite below). Must be
   green before step 6.

6. **TestPyPI dry-run.**
   - `poetry build`
   - **Wheel-version sanity check** (issue #131). Catches the
     0.4.2-style hardcoded-version drift before the wheel ever
     reaches TestPyPI:
     ```bash
     WHEEL=$(ls dist/almaapitk-*-py3-none-any.whl)
     WHEEL_VER=$(echo "$WHEEL" | sed -E 's/.*almaapitk-([0-9.]+)-py3.*/\1/')
     PYPROJECT_VER=$(grep '^version' pyproject.toml | sed -E 's/version = "([^"]+)"/\1/')
     [ "$WHEEL_VER" = "$PYPROJECT_VER" ] || { echo "wheel/pyproject version mismatch: $WHEEL_VER vs $PYPROJECT_VER" >&2; exit 2; }
     ```
   - `poetry publish -r testpypi`
   - In a throwaway venv: `pip install -i https://test.pypi.org/simple/ almaapitk==0.4.0`
   - Smoke import: `python -c "from almaapitk import AlmaAPIClient, Configuration, Admin, Users, Acquisitions, ResourceSharing, Analytics; print('OK')"`
   - If anything looks wrong, fix on `release/0.4.0` and re-run from
     step 5; do NOT publish to real PyPI yet.

7. **Publish to PyPI.**
   - Open a PR from `release/0.4.0` into `main` and merge it.
   - `git checkout main && git pull`.
   - `poetry publish` (irreversible).

8. **Tag and GitHub Release.**
   - `git tag -a v0.4.0 -m "Release 0.4.0"`
   - `git push origin v0.4.0`
   - `gh release create v0.4.0 --title "v0.4.0" --notes-file <(awk '/^## \[0.4.0\]/,/^## \[0.3.1\]/' CHANGELOG.md | head -n -1)`
     (or paste excerpt manually)

9. **Post-release smoke.**
   Fresh venv, `pip install almaapitk==0.4.0`, run the same import smoke.

## Pre-release scope

**Hard prerequisites before step 7:**

- `users-requests` PR merged to `main` (otherwise the Users coverage
  block in the changelog has a dangling promise).
- `ALMA_PROD_API_KEY` unset in the publish shell (R8 enforcement
  blocks `chunks regression-smoke` otherwise).

**Explicitly out of scope:**

- GitHub Actions publish workflow — filed as a follow-up issue after
  `0.4.0` ships. (Landed post-0.4.3 via issue #128:
  `.github/workflows/release.yml` is now the canonical publish path on
  tag push; manual `poetry publish` / `twine upload` becomes the
  fallback documented in `docs/RELEASE_CHECKLIST.md` Phase J.)
- Remaining open architecture issues (`#3–#21`) — ship in `0.5.0` or
  later. Don't expand scope mid-release.

## Validation suite

Run on `release/0.4.0` after the version bump and CHANGELOG finalize.
Any failure stops the release.

1. `poetry install --sync` — lock file resolves cleanly under current
   Python.
2. `poetry run python scripts/smoke_import.py` — public-API smoke.
3. `poetry run pytest tests/test_public_api_contract.py -v` — locks
   the import surface.
4. `poetry run pytest tests/unit/ tests/logging/ tests/integration/client/ -v`
   — full unit + logging + client integration suites.
5. `unset ALMA_PROD_API_KEY && scripts/agentic/chunks regression-smoke`
   — R10 cumulative bug-driven regression suite.
6. `poetry build` — produces `dist/almaapitk-0.4.0-py3-none-any.whl`
   and `.tar.gz`. Inspect via `unzip -l dist/*.whl` to confirm only
   `almaapitk/` package code ships — no `tests/`, `scripts/`, `docs/`,
   `.a5c/`, `logs/`, `config/`, no `CLAUDE.md`/`AGENTS.md`. The
   `pyproject.toml` exclude list already covers this; the inspection
   is verification.

## Rollback plan

PyPI does not allow re-uploading the same version. If `0.4.0` is
broken on PyPI after step 7, the recovery is `0.4.1`:

1. Yank `0.4.0` via the PyPI web UI to discourage installs (does not
   delete; pinned consumers still get it).
2. Fix on `main`.
3. Re-run the full validation suite.
4. Publish `0.4.1`.

This is the same path that `0.3.0 → 0.3.1` followed. The TestPyPI
gate in step 6 is the guard that should make this nearly never
needed.

## Consumer impact

The release does not affect existing installs:

- Already-installed `0.3.1` environments: zero effect.
- Fresh installs / explicit `pip install -U`: resolve to `0.4.0`.
- `==0.3.1` pins: safe.
- `~=0.3.1` (i.e., `>=0.3.1,<0.4.0`): safe.
- `>=0.3.1` or unpinned: could resolve to `0.4.0` on next install.
- Lockfile users (`poetry.lock`, `uv.lock`, `pip-tools`): safe until
  they explicitly refresh the lock.

Since `0.4.0` is purely additive, an upgrade is low-risk for any
consumer that takes it. The four extracted consumer repos
(`Alma-update-expired-users-emails`,
`Alma-RS-lending-request-automation`, `Alma-Digital-Upload`,
`Alma-Acquisitions-Automation`) each declare `almaapitk` somewhere;
whether they auto-pick up `0.4.0` on their next deploy depends on how
each pinned it. Reviewing those pins is out of scope for this
release.
