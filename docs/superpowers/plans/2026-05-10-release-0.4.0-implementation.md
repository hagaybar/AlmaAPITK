# Release 0.4.0 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Cut and publish `almaapitk 0.4.0` to PyPI, gated by a TestPyPI dry-run, with a `release/0.4.0` branch carrying the version bump and CHANGELOG finalize as a single reviewable unit.

**Architecture:** Linear release pipeline. Nothing is irreversible until Task 8 (`poetry publish` to real PyPI). All earlier tasks live on `release/0.4.0` and can be discarded by deleting the branch. Manual mechanics throughout — no CI publishing on this release.

**Tech Stack:** Python 3.12, Poetry 2.x for packaging, `gh` CLI for GitHub operations, PyPI + TestPyPI for distribution.

**Spec:** `docs/superpowers/specs/2026-05-10-release-0.4.0-design.md`

```bash

```

Expected: every chunk's SANDBOX smoke tests pass against current `main` (well, `release/0.4.0` — same code). If any chunk fails, triage:

- **R8 violation** (`R8: orchestration env must be SANDBOX-only`): prod key is set; the `unset` above should have cleared it. Re-run with `unset` chained on the same line.

- **Real regression** (assertion failure, `AlmaAPIError` in code paths that worked before): block the release; investigate and fix on `release/0.4.0`.

## Task 5: TestPyPI dry-run

**Files:** None (uses dist artifacts from Task 4)

- [ ] **Step 5.1: Verify TestPyPI repository is configured in poetry.**

```bash

```

Expected: prints `https://test.pypi.org/legacy/` (or similar TestPyPI URL).

If it errors with `[ValueError] No such repository`, configure it once:

```bash

```

- [ ] **Step 5.2: Verify TestPyPI auth token is set.**

This is a best-effort check — `poetry` does not print the stored token, so the only authoritative test is the `poetry publish -r testpypi` call in Step 5.3. To set the token if you don't have it stored:

```bash
poetry config pypi-token.testpypi <YOUR_TESTPYPI_TOKEN>
```

(Get the token from https://test.pypi.org/manage/account/token/. NEVER commit the token. If Step 5.3 fails with `401 Unauthorized`, set or refresh the token and retry.)

- [ ] **Step 5.3: Upload to TestPyPI.**

```bash
poetry publish -r testpypi
```

Expected: uploads `dist/almaapitk-0.4.0-py3-none-any.whl` and `dist/almaapitk-0.4.0.tar.gz`. Final line says something like `Publishing almaapitk (0.4.0) to TestPyPI` followed by upload progress.

If TestPyPI rejects with `File already exists` for `0.4.0`, you've already uploaded this version once — skip to step 5.4 unless you've changed the wheel since (in which case bump the local version to `0.4.0.post1`, re-build, re-publish to TestPyPI, then revert to `0.4.0` for the real PyPI publish).

- [ ] **Step 5.4: Wait ~30 seconds for TestPyPI to index, then verify the page.**

Open https://test.pypi.org/project/almaapitk/0.4.0/ in a browser. Confirm:
- Version `0.4.0` listed as latest
- README renders without errors (check for "Description content type" warnings)
- All `[project.urls]` links present (Homepage, Repository, Documentation, Issues, Changelog)

- [ ] **Step 5.5: Create a throwaway venv and install from TestPyPI.**

```bash
TMPVENV=$(mktemp -d)/venv
python3.12 -m venv "$TMPVENV"
source "$TMPVENV/bin/activate"
pip install --upgrade pip
pip install -i https://test.pypi.org/simple/ \
  --extra-index-url https://pypi.org/simple/ \
  almaapitk==0.4.0
```

The `--extra-index-url https://pypi.org/simple/` is required because TestPyPI does not mirror runtime dependencies (e.g., `requests`); pip falls back to real PyPI for those.

Expected: `Successfully installed almaapitk-0.4.0` and `requests-X.Y.Z` (and `requests` transitives).

Expected: `IMPORT OK 0.4.0`. No `ImportError`, no `AttributeError`.

- [ ] **Step 5.7: Tear down the throwaway venv.**

If steps 5.5 or 5.6 failed, **stop here.** Fix the issue on `release/0.4.0`, re-run from Task 4 (build), and re-attempt TestPyPI. Do NOT continue to Task 6 until TestPyPI install + smoke is green.

## Task 10: File follow-up issue and close out

**Files:** None (gh issue create)

- [ ] **Step 10.1: File the GitHub Actions publish-workflow follow-up.**

```bash
gh issue create --title "Add GitHub Actions publish workflow (PyPI Trusted Publishing on tag push)" --body "$(cat <<'EOF'
## Background

`0.4.0` was published manually with `poetry build` + `poetry publish`,
matching `0.3.1`'s flow. Future releases should automate the publish
step.

## Proposal

Add a `.github/workflows/release.yml` that:

- Triggers on `push` of any `v*` tag.
- Runs the standard validation suite (smoke import, public API contract,
  unit + logging + client integration suites).
- Builds the wheel + sdist.
- Publishes to PyPI via [Trusted Publishing](https://docs.pypi.org/trusted-publishers/)
  using OIDC — no token in repo secrets.

## Prerequisites

1. Configure the PyPI project as a Trusted Publisher pointing at this
   repository + workflow + tag environment.
2. (Optional) Create a separate "release" environment in GitHub repo
   settings for additional approval gates.

## Out of scope

- Migrating to a different build tool.
- Auto-bumping the version (manual bump remains the trigger).

## Acceptance

- Pushing a `v*` tag triggers the workflow, runs validation, and
  publishes to PyPI without manual `poetry publish`.
- Documented in `docs/superpowers/specs/` once implemented.
EOF
)" --label enhancement
```

Expected: prints the new issue URL.

- [ ] **Step 10.2: Sanity-check the final state.**

## Rollback plan (if Task 7 publishes a broken artifact)

PyPI does not allow re-uploading the same version. Recovery:

1. **Yank `0.4.0`** via the PyPI web UI (https://pypi.org/manage/project/almaapitk/release/0.4.0/) — discourages new installs but preserves it for pinned consumers.
2. **Fix on `main`.**
3. **Re-run Tasks 2 (CHANGELOG add a `[0.4.1]` section), 3 (bump version to `0.4.1`), 4 (validation), 5 (TestPyPI), 6 (PR), 7 (publish), 8 (tag `v0.4.1`), 9 (smoke).**
4. Same path that `0.3.0 → 0.3.1` followed.

The Task 5 TestPyPI gate is the guard that should make this nearly never needed.
