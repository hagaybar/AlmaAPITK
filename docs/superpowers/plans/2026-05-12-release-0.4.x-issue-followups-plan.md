# Release 0.4.x Review — Issue Follow-Ups Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Designed for babysitter execution — every step is a discrete idempotent shell command.

**Goal:** File three comments on existing GitHub issues and create eight new GitHub issues that follow up on the findings of the 2026-05-12 0.4.x release review, then update the spec with assigned issue numbers.

**Architecture:** Pure orchestration. No source-code changes. Every step is a `gh` CLI invocation guarded by an idempotency check (skip-if-already-posted for comments, skip-if-title-already-exists for new issues). The bodies are pre-authored verbatim in the spec doc; this plan re-uses them via heredoc.

**Tech Stack:** `gh` CLI (≥ 2.x), `jq`, bash. Single-machine, no concurrency, no rate-limit concerns (GitHub allows ~5000 calls/hour for authenticated users; we make ≤ 30).

## Idempotency strategy

Every step that posts to GitHub embeds a unique HTML-comment marker in the posted body. Pre-check scripts grep for the marker via `gh issue view --comments` (for comments) or `gh issue list --search` (for new issues). If found, the step is a no-op. This lets a babysitter re-run pick up where it left off if a step failed mid-flight.

Markers used:
- Comments: `<!-- release-0.4.x-review-followup:<issue#> -->`
- New issues: `<!-- release-0.4.x-review-followup:issue-<letter> -->` in the body (titles cannot carry HTML comments, so the check pivots on title-exact-match).
