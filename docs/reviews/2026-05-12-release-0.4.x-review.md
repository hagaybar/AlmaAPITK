## Executive Summary

The v0.3.1..HEAD range is a coherent 0.4.x release cycle. The chunk-driven pipeline shipped substantial new HTTP infrastructure (persistent session, single `_request` chokepoint, retry, configurable timeout/region, typed `AlmaAPIError` subclasses, `iter_paged` generator, context-manager support, `AlmaResponse` memoization), a brand-new Configuration domain with 35 methods, large Users-domain coverage (loans/requests/CRUD/search/personal data), and Admin Sets full CRUD. Public-API additions are exposed via `__all__` and the lazy-import layer is intact; new domain methods are accompanied by ~221 unit tests in `test_configuration.py` and ~285 in `test_users.py`.

The four-bump release history (0.4.0 → 0.4.3, with 0.4.2 yanked) was driven by two distinct gaps the validation suite missed: (a) top-level discoverability surfaces (README table, `docs/index.md`, `docs/getting-started.md`) did not mention the new Configuration / Analytics / typed-exception symbols, and (b) `src/almaapitk/__init__.py` held a hardcoded `__version__` literal that drifted from `pyproject.toml`. Both are now patched, with a regression test (`tests/test_version.py`) and a binding `RELEASE_CHECKLIST.md`. Substantive code-quality and release-hygiene issues remain: `docs/index.md` and `docs/getting-started.md` still show "Version: 0.2.0"; the README API-reference table (lines 167-176) still omits Configuration and all six typed `AlmaAPIError` subclasses (the Phase C checklist only enforces the bullet list, not the table); a bare `except:` at `acquisition.py:1814` survived the client-ergonomics chunk's narrow-except cleanup; `Users.__init__` tears down the project `alma_logging` contract and re-creates a console+file logger that writes `sb_log_file.log` / `prod_log_file.log` into the caller's CWD on every instantiation; and the bug-fix for t-41-3 (#41) and the `update_letter` XML fix (#114) each predate full R10 regression-test coverage in the cumulative suite. Overall severity HIGH: nothing is data-lossy or security-critical, but a future release will repeat the four-bump pattern unless Phase C is hardened beyond the bullet list it currently inspects.

## What Worked

- Chunk-driven pipeline shipped 19+ chunks over ~2 weeks with a coherent commit history, manifest-based traceability (`Refs #N` in every commit message), and a generated `AGENTIC_RUN_LOG.md` that captures pass/fail counts per chunk.
- Public API contract preserved: `get`/`post`/`put`/`delete` signatures bit-for-bit identical; `AlmaResponse` cache is transparent; `iter_paged` is opt-in; typed `AlmaAPIError` subclasses inherit so `except AlmaAPIError:` still catches everything.
- R10 (bug-driven regression tests) was honoured for the `__version__` drift bug — `tests/test_version.py` was committed alongside the fix in 2479489 and reproduces the failure on the broken tree.
- R9 (no real IDs in committed content) was honoured for chunk artifacts: `chunks/*/test-data.json` is gitignored, sandbox-tests load fixtures at runtime, prompt templates use synthetic placeholders.
- `AlmaAPIClient` rewrite from per-verb HTTP to a single `_request` chokepoint is clean and well-documented; the persistent Session, retry policy, region/host configurability, and context-manager support all land without breaking the `AlmaResponse` wrapper contract.
- `CHANGELOG.md` is candid about the four-bump cycle and explains the yank in plain language — operators upgrading from 0.4.2 will understand what to do.

## What Failed

- The 0.4.0 release went out with three independent gaps in the top-level docs (README table, `docs/index.md`, `docs/getting-started.md`) all missing Configuration / Analytics / typed exceptions; the TestPyPI gate caught these one-at-a-time across three bumps because each fix was scoped narrowly to the file the operator opened.
- `src/almaapitk/__init__.py` held a hardcoded `__version__` literal that the version-bump step in `pyproject.toml` never touched — a structural sin that survived multiple chunk PRs, every release-validation checklist run, and made it onto PyPI before the post-publish smoke caught it.
- Bug fixes (#114 `update_letter` XML body, #41 availability-cache race) shipped without R10 unit-test counterparts — the smoke test exists but the bug-driven discipline of "failing test FIRST, in same commit as fix" was not consistently applied for these two.
- `Users.__init__` pre-existing legacy logger setup (CWD file writes, console handler) was not refactored despite being touched by five separate chunks (#36, #37, #39, #40, #41, #44, #45) that all instantiated `Users`.
- `RELEASE_CHECKLIST.md` is now binding but is markdown-only — the same human-fallibility surface that caused the four-bump pattern. The post-mortem identified the right anti-patterns but did not convert them into machine checks.

## The One Process Change

**Single process change that would have prevented the four-bump pattern:**

Convert `RELEASE_CHECKLIST.md` Phase C (docs completeness) and Phase E (version drift) from human-checkbox items into pytests under `tests/meta/`. Three concrete tests: (1) `test_top_level_docs_match_all.py` — parses `__all__` from `src/almaapitk/__init__.py` and asserts every symbol appears in `README.md` AND `docs/index.md` AND `docs/getting-started.md`; (2) `test_no_hardcoded_version.py` — ast-walks `src/almaapitk/` and fails on any `__version__ = '<string-literal>'` assignment (the `test_version.py` runtime check is necessary but not sufficient); (3) `test_docs_version_matches_pyproject.py` — greps every `Version: X.Y.Z` heading in `docs/**/*.md` and asserts it equals `pyproject.toml`'s version. With these three tests wired into Phase F (the pytest gate), the four-bump pattern becomes impossible: a future operator who skips the manual Phase C still gets a failing test at Phase F, and the release stops before TestPyPI ever sees the bad artifact. The 0.4.x cycle taught the right lessons; the process change is to make those lessons enforced by pytest rather than by operator vigilance.

### F-017 — LOW — test-coverage
**Summary.** `scripts/agentic/regression_smoke.py` and the R10 test discipline are documented in `CLAUDE.md` and the playbook, but the cumulative regression-smoke run is not wired into the standard validation suite — it's an optional gate the operator must remember to run.

**Evidence:** `docs/RELEASE_CHECKLIST.md:67` lists `scripts/agentic/chunks regression-smoke` as a Phase F item but with the soft acceptable-exception clause for env-blocked chunks. Without a JSON-reported pass/fail this is a human-judgement gate. `CLAUDE.md` R10 establishes the discipline but doesn't say where the resulting tests live (some are in `tests/unit/`, some in `chunks/*/sandbox-tests/`) — risk of bug-fix tests being filed in a `chunks/` directory that gets stale if the chunk is later aborted/abandoned.

**Recommendation:** Specify the home for R10 unit tests: bug-driven regressions should live in `tests/unit/regressions/` (a new directory) named `test_issue_<N>.py`, not inside `chunks/<name>/`. Chunks artifacts can keep their SANDBOX smokes; the unit regressions are the canonical regression suite. This makes R10 grep-able and survives chunk lifecycle transitions.

### F-018 — LOW — api-risk
**Summary.** `AlmaResponse.data` is documented to memoize but the underlying `_safe_body()` path does NOT cache when the parse returns `None` (non-JSON or malformed body), so callers reading `r.data` in a tight loop on a non-JSON response re-traverse content-type headers every iteration.

**Evidence:** `src/almaapitk/client/AlmaAPIClient.py:140-144` — 'Only cache successful parses. A None result either means "not JSON content-type" (cheap to recompute) or "malformed JSON" (we want a later `.json()` to still raise rather than silently return None from cache).' The reasoning is sound for the malformed-JSON case but unnecessarily skips caching the cheap-but-not-free 'content-type not JSON' decision.

**Recommendation:** Split the cache: cache the parsed body OR the content-type-rejection sentinel separately, so that the non-JSON path is also single-shot. Or accept that 'cheap to recompute' was already the reviewer's intent and document this in the `.data` property docstring rather than only in `_safe_body()`. Very low priority — the cost difference is microseconds per call.

### F-019 — LOW — code-quality
**Summary.** `Configuration.list_libraries` (and many sibling `list_*` methods) pass `params={'limit': '100', 'offset': '0'}` as strings rather than the `iter_paged`-style ints; combined with the explicit comment 'we do not need `iter_paged` here', this is a subtle inconsistency with the rest of the codebase.

**Evidence:** `src/almaapitk/domains/configuration.py:246-249` and recurring throughout `configuration.py`. `iter_paged` passes them as ints (`AlmaAPIClient.py:1068-1072`). Alma accepts both string and int per the API, so this is cosmetic — but reviewers grepping for `limit=100` (int) miss these call sites.

**Recommendation:** Pass ints, not strings, in `configuration.py`: `params={'limit': 100, 'offset': 0}`. Or — preferable — use `iter_paged` uniformly with a `max_records` cap of 100 even where the operator expects 'small list'. Defensive against the day an institution has 101 libraries.

## Methodology

This review covers `git log v0.3.1..HEAD` — 175 commits across the v0.4.0, v0.4.1, v0.4.2 (yanked), and v0.4.3 releases plus two post-release docs commits. The range is treated as one logical release because all four version bumps shipped the same chunk-driven feature set; the splits were forced by release-process mistakes (top-level docs gaps caught on TestPyPI and a hardcoded `__version__` literal caught on PyPI), not by independent feature work. Findings are organized along six review dimensions — code-quality, api-risk, test-coverage, release-hygiene, security, and regression-smell — chosen to surface both the technical surface area shipped and the process surface area that produced the four-bump pattern. Severity uses a four-step rubric (CRITICAL / HIGH / MEDIUM / LOW): CRITICAL is data-loss or active security exposure, HIGH is a defect that will recur on the next release or that already misleads operators in production, MEDIUM is a code-quality or release-hygiene issue that compounds over time, LOW is cosmetic or pre-existing without near-term impact. The review was performed by a Claude Code subagent (`superpowers:code-reviewer`) driven by a babysitter orchestration process; the structured JSON findings file and intermediate artifacts are preserved under the babysitter run directory at `.a5c/runs/01KRD99E809AFSF3XE95GH27ZP/`.
